Contrat de données : la Vente

Modèle canonique d'une vente de l'Atelier ALM, prêt à mapper sur des tables Supabase. Chaque champ, son type, son enum, sa nullabilité et l'écran analytique qui le consomme. Ce document fige le contrat ; la structure des tables reste à l'appréciation de CC. v1 — révisable sur deux fronts : (1) après réception d'un export caisse boutique réel (mapping d'import et déduction de mode provisoires), et (2) les corrections de conformité actées avec l'équipe — emplacement = référentiel éditable à FK (plus un enum), et modèle à 2 modes unite·poids (mode portion et taille S/M/L retirés).

01Le modèle vente

Un enregistrement par transaction encaissée (ou devis, pour le traiteur). Les lignes de panier vivent dans une table enfant vente_ligne (§02). Le champ fulfillment pilote la répartition entre file de production et historique.

Règle CA — critique

Le CA (et les KPI Finances / Historique) ne se comptent que sur fulfillment='remis'. Une commande traiteur pour vendredi saisie aujourd'hui n'entre pas dans le CA du jour de saisie — elle y bascule à la remise. La page Commandes du jour n'affiche que le non-remis.

02Décision : le bowl composé — « les deux en couches »

Un bowl truck s'enregistre comme une ligne recette virtuelle (type='bowl') qui référence ses composants. Le modèle porte les deux niveaux simultanément :

Niveau produit → C1
L'analytique commerciale lit le bowl comme un produit unique — « le coco-curry cartonne le mercredi ». Le recette_id virtuel porte le nom du bowl.
Niveau composant → C2
L'analytique cuisine déplie vers les 4 composants (protéine·féculent·légume·sauce) pour le gaspi et le coût matière. Chaque composant = une sous-ligne.
-- Une ligne de vente = un bowl (recette virtuelle, se déplie),
-- un produit fini, ou une formule traiteur. Tarification : 2 modes.
create table vente_ligne (
  id            uuid primary key,
  vente_id      uuid references vente(id),
  type          ligne_type,        -- bowl | produit | formule
  mode          ligne_mode,        -- unite | poids  (2 modes)
  recette_id    uuid references recette(id), -- bowl = recette VIRTUELLE (se déplie)
  poids_g       int,               -- mode=poids UNIQ. — pesée réelle (continu), sinon NULL
  prix_kg       numeric(6,2),      -- mode=poids UNIQ. — prix au kilo, sinon NULL
  qte           int,               -- mode=unite — pièces ; NULL en poids
  prix_unitaire numeric(6,2),      -- mode=unite ; NULL en poids
  montant       numeric(8,2)       -- unite: PU×qte · poids: prix_kg×poids_g/1000
);

-- Dépliage du bowl → 4 composants (coût matière + gaspi C2).
-- Le bowl se VEND à l'unité mais reste COMPOSÉ. Vide pour un produit fini.
create table vente_ligne_composant (
  ligne_id      uuid references vente_ligne(id),
  composant_id  uuid references composant(id),
  categorie     categorie        -- proteine|feculent|legume|sauce
);
Champ mode — 2 tarifications. unite (prix_unitaire × qté) : bowls truck en format unique, plats préparés, plats/plateaux traiteur, produits boutique à la pièce. poids (poids_g × prix_kg) : produits boutique à la pesée. Plus de mode portion ni de champ taille S/M/L — supprimés du modèle. Le bowl reste composé : vendu à l'unité, il se déplie en 4 composants via vente_ligne_composant (couche cuisine = coût matière + gaspi C2). « Les deux en couches » : couche produit = unite, couche cuisine = composants.
Catalogue — 3 catalogues de vente distincts + 1 couche composants commune. Schéma produit : nom, categorie, canal (truck|boutique|traiteur), mode (unite|poids), prix selon mode (prix_unitaire ou prix_kg), dispo. Les catalogues de vente sont propres à chaque canal, peu de recouvrement (bowls & plats truck ; plats du jour au poids + parts/pains à l'unité en boutique ; plats & plateaux traiteur). Exception : la couche composant (poulet, brocoli, riz…) est COMMUNE sous les catalogues — un composant est une brique de bowl truck ET peut être un produit au poids en boutique. Catalogues de vente distincts, couche composants partagée (coût matière + gaspi). Pas de silos étanches.
À trancher côté base : la recette virtuelle d'un bowl est-elle matérialisée en ligne dans recette (une par combinaison signature vendue), ou générée à la volée depuis les 4 composants ? Recommandation : une recette virtuelle par bowl signature au menu (le coco-curry existe comme produit), les compositions libres restant dépliées sans recette parente.

03Le timestamp n'est pas un détail

occurred_at — timestamptz — NOT NULL

Dans la maquette, « aujourd'hui » est codé en dur. En base, c'est ce champ qui porte le jour-de-semaine, et donc TOUTE la saisonnalité : le pic du mercredi (Ventes & tendances), le prisme de demande de la Production, la courbe CA par jour. Il doit être capturé à l'encaissement, en fuseau Europe/Paris, et ne jamais être dérivé de created_at technique (une vente saisie en différé garde sa date réelle).

05Import de caisse — fin de journée (boutique)

La boutique dispose d'une caisse/balance qui exporte un fichier en fin de journée. L'écran Import caisse reconstitue les ventes du jour par lot au lieu de les ressaisir. Flux : dépôt du fichier → mapping des colonnes → prévisualisation → validation en batch. Chaque ligne devient une vente boutique (mode_vente='instantane' → fulfillment='remis'), avec ses vente_ligne en mode poids ou unite (§02).

Mapping configurable — à caler
Le parseur n'est PAS figé sur un format supposé. Le mapping colonne CSV → champ modèle est paramétrable (détection du séparateur ; , tab). Format réel de la caisse ALM pas encore fourni — à caler sur un vrai export avant prod. Déduction du mode provisoire : faute de colonne « mode » dans le fichier, le mode (poids/unité) est deviné (kg→poids, qté→unité) et chaque ligne est signalée « à confirmer » dans la prévisualisation — à remplacer par un mapping de mode explicite. Lignes non rapprochées exclues du batch ; les ventes créées sont marquées source='import' (distinctes d'une saisie manuelle).
Désignation → recette_id / composant
Poids (kg) → poids_g (×1000)
Quantité → qte
Montant → montant
Règlement → moyen_paiement
Horodatage — critique
occurred_at = jour de l'import (date réelle), sans heure précise — acceptable pour la boutique. Ne jamais dériver occurred_at d'un created_at technique : la date d'insertion en base ≠ la date de vente réelle. Reporter la date métier du fichier / du jour d'exploitation.
Lignes non rapprochées (désignation inconnue, montant manquant) signalées et exclues du batch jusqu'à correction du mapping — jamais créées à l'aveugle. Validation = création groupée des seules lignes valides. Démo — parseur sur CSV d'exemple.

06Figé vs. à décider côté base

✓ Figé — ne pas rediscuter
  • Le jeu de champs de vente et leur nullabilité (§01).
  • Les enums et leurs clés exactes (§04) — dont mode_vente, marqueur qui dérive fulfillment.
  • Le bowl = recette virtuelle + référence composants, deux niveaux (§02).
  • emplacement obligatoire pour le truck, NULL ailleurs.
  • Le paiement remplace le statut d'encaissement (l'Atelier est un cahier de vente, pas une caisse).
  • occurred_at = jour de vente réel ; à l'import caisse = jour d'import, jamais un created_at technique (§05).
? À décider — appel base (CC)
  • Matérialisation de la recette virtuelle bowl (§02) — reco donnée, à valider.
  • Normalisation des enums : types Postgres create type vs. tables de référence + FK.
  • Dérivation de montant_total (stocké vs. calculé à la lecture).
  • Rattachement multi-emplacements futur (marchés supplémentaires) — la liste PLACES est déjà paramétrable.
  • Format réel du CSV caisse (§05) : mapping à caler sur un vrai export avant prod.
  • Indexation pour la saisonnalité (index sur occurred_at, canal).
Référence maquette : écran Saisie de vente (source de vérité du contrat). Les écrans consommateurs listés en §01 lisent ces champs tels quels — aucune transformation cachée côté front.
